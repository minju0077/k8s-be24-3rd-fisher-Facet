# 💳 결제 및 정산 (Payment) 상세 시스템 아키텍처 및 핵심 기능 명세

> FACET 서비스의 자금 흐름을 담당하는 결제 및 정산 시스템입니다. 유저의 포인트 충전부터 펀딩/경매 결과에 따른 자동 결제까지, 클라이언트 데이터를 신뢰하지 않는(Zero-Trust) 서버 중심의 검증과 스케줄링 아키텍처를 구현했습니다.

---

## 1. 💰 유저 포인트 충전 및 2단계 무결성 검증 (Point Charge & Verification)
PortOne SDK를 연동하여 카드, 간편결제 등 다양한 결제 수단을 제공하며, 프론트엔드 조작을 원천 차단하기 위한 서버 사이드 교차 검증을 수행합니다.

### ⚙️ 어떻게 구현했는지 (How)
* **공식 Java SDK 도입:** `io.portone.sdk.server` 패키지를 활용해 `PaymentClient`를 Spring Bean으로 등록하고(DI), API Secret은 `@Value("${PORTONE_SECRET}")` 환경변수로 은닉화했습니다.
* **2단계 교차 검증 로직:** 1. **사전 기록:** 백엔드가 먼저 '충전 예정 금액'을 DB에 가주문 상태로 기록합니다.
    2. **영수증 대조:** 결제 완료 후, 프론트가 보내주는 금액을 믿지 않고 PortOne 서버에 `paymentId`(영수증 번호)로 직접 API를 호출해 **실제 결제 금액**을 가져와 DB의 예정 금액과 일치하는지 검증합니다.
* **어뷰징 방지:** 결제 영수증 번호(`pgPaymentId`)를 DB에 고유값(Unique)으로 저장하여 중복 처리를 막고, 2단계 검증을 통과한 주문(`paid=true`)에 한해서만 포인트를 적립합니다.

### 💡 왜 이렇게 했는지 (Why)
* **프론트엔드 위변조 방지 (Zero-Trust):** 브라우저의 개발자 도구를 통해 결제 금액을 임의로 조작(예: 10,000원을 100원으로 결제 요청)하는 악의적인 시도를 막기 위해, 모든 핵심 데이터는 **백엔드 서버와 PG사 서버 간의 직접 통신(S2S)**을 통해서만 확인하도록 강력한 보안을 적용했습니다.

## 2. 🔑 PortOne V2 빌링키(Billing Key) 발급 및 예약 결제
사용자가 펀딩에 참여하거나 경매에 입찰할 때, 당장 돈이 빠져나가지 않고 결제 수단만 등록해 두는 '예약 결제' 시스템을 구축했습니다.

### ⚙️ 어떻게 구현했는지 (How)
* **빌링키 발급 API 연동:** 사용자가 카드 등록 창에서 인증을 마치면, 실제 결제를 일으키는 대신 나중에 결제할 수 있는 권한인 **'빌링키(Billing Key)'**를 PG사로부터 발급받아 암호화하여 DB에 보관합니다.
* **가주문(Pending) 상태 관리:** 결제가 즉시 이루어지지 않으므로 주문 테이블에 `PAYMENT_PENDING` (결제 대기) 상태로 주문서를 생성하고 사용자의 입찰 및 펀딩 참여를 확정합니다.

### 💡 왜 이렇게 했는지 (Why)
* **보안 리스크 완벽 차단:** 사용자의 신용카드 번호 같은 민감한 금융 정보를 서비스 DB에 직접 저장하는 것은 심각한 보안 리스크가 따릅니다. PG사가 안전하게 관리하는 빌링키만 보관하고, 지정된 날짜에 서버 간 통신으로 결제를 요청하도록 설계하여 보안성과 법적 컴플라이언스를 준수했습니다.

## 3. ⏱️ Spring Scheduler 기반 조건부 자동 결제 및 취소
펀딩 마감일이나 경매 종료 시점에 관리자의 개입 없이, 시스템이 스스로 조건을 판단하여 결제 또는 취소를 수행합니다.

### ⚙️ 어떻게 구현했는지 (How)
* **스케줄링 배치 작업:** Spring의 `@Scheduled` 어노테이션을 활용하여 매일 자정(또는 지정된 시각)에 종료된 프로젝트를 조회하는 배치 로직이 실행됩니다.
* **조건부 결제 승인 (Pay):** 펀딩이 100% 이상 달성되었거나 경매 낙찰자가 확정된 경우, 보관 중이던 '빌링키'를 사용해 PortOne 서버로 결제 승인 API를 호출합니다. 승인 완료 시 주문 상태를 `PAID`로 업데이트합니다.
* **일괄 자동 취소 (Cancel):** 펀딩 목표 금액 미달 시, 시스템이 예약된 가주문 내역을 일괄적으로 `CANCELLED` 처리하고 관련된 리소스를 안전하게 해제합니다.

### 💡 왜 이렇게 했는지 (Why)
* **운영 리소스 최소화 및 휴먼 에러 방지:** 수백 건의 결과를 관리자가 일일이 확인하고 수동으로 결제하는 것은 불가능합니다. 이를 자동화된 스케줄러로 위임하여 비즈니스 로직의 신뢰성을 높이고, 정확한 시점에 약속된 결제와 취소가 이루어지도록 보장했습니다.

## 4. 🛡️ 서버 간 통신(S2S) 예외 처리 및 트랜잭션 관리
외부 PG사 서버와의 통신 중 발생할 수 있는 다양한 실패 시나리오를 대비하여 데이터 정합성을 지킵니다.

### ⚙️ 어떻게 구현했는지 (How)
* **에러 코드 핸들링:** 잔액 부족, 한도 초과, 유효기간 만료 등의 PG사 에러 코드를 세분화하여 핸들링합니다.
* **상태 동기화 및 롤백:** 결제 승인 도중 네트워크 타임아웃이나 예외가 발생할 경우, DB의 주문 상태를 `PAYMENT_FAILED`로 즉시 변경하고 트랜잭션을 안전하게 롤백(Rollback)합니다.

### 💡 왜 이렇게 했는지 (Why)
* **Design for Failure (실패를 고려한 설계):** 외부 API 통신은 언제든 실패할 수 있습니다. 우리 DB는 '결제 완료'인데 PG사는 '결제 실패'인 상태(데이터 불일치)가 되면 심각한 금전적/CS 이슈가 발생합니다. 트랜잭션 경계를 명확히 하고, 실패 시 안전하게 복구되도록 방어 로직을 구성하여 결제 시스템의 안정성을 극대화했습니다.
