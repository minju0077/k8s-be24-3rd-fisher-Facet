# 👤 회원 및 권한 (User) 상세 시스템 기능 명세

> FACET 서비스의 기반이 되는 유저 인증, 인가, 그리고 안전한 포인트 결제 시스템의 아키텍처를 다룹니다. 클라이언트의 데이터를 절대 신뢰하지 않는(Zero-Trust) 백엔드 중심의 철저한 보안과 교차 검증 로직에 초점을 맞추어 설계했습니다.

---

## 1. 💳 유저 포인트 충전 및 결제 무결성 검증 (Point Charge & Payment Integrity)
다양한 결제 수단을 제공하는 PortOne(포트원) SDK를 연동하여 포인트 충전 기능을 구현했으며, 악의적인 결제 금액 조작을 원천 차단하기 위한 서버 사이드 검증을 수행합니다.

### ⚙️ 어떻게 구현했는지 (How)
* **단일 SDK 및 환경변수 관리:** 카드, 간편결제 등을 통합 처리하는 PortOne SDK를 도입했으며, API Secret Key는 `@Value("${PORTONE_SECRET}")` 환경변수로 분리하여 소스 코드 노출을 방지했습니다.
* **PaymentClient DI 주입:** 공식 Java SDK의 `PaymentClient`를 Spring Bean으로 등록해 서비스 레이어에서 의존성 주입(DI)으로 사용합니다.
* **2단계 교차 검증 로직:** 1. **사전 기록:** 백엔드가 먼저 '충전 예정 금액'을 DB에 기록합니다. (가주문 생성)
    2. **영수증 대조:** 결제 완료 후 프론트가 전달하는 금액을 믿는 대신, PortOne 서버에 `paymentId`(영수증 번호)로 직접 API를 쏴서 **실제 결제 금액**을 가져와 DB의 예정 금액과 대조합니다.

### 💡 왜 이렇게 했는지 (Why)
* **프론트엔드 조작 방지 (Zero-Trust):** 웹 브라우저의 개발자 도구나 네트워크 탭을 통해 사용자가 결제 금액을 변조(예: 10,000원 충전을 100원으로 결제)할 위험이 있습니다. 포트원 서버와의 직접 통신(S2S)을 통해 금액을 검증함으로써 조작 시도를 즉시 탐지하고 차단합니다.
* **이중 결제 및 어뷰징 방지:** 결제 영수증 번호(`pgPaymentId`)를 DB에 고유값으로 저장하여 동일한 영수증을 재사용하는 어뷰징을 막고, 2단계 검증을 완벽히 통과한 주문(`paid=true`)에 한해서만 포인트를 적립하도록 안전하게 설계했습니다.

## 2. 🔐 통합 인증 및 소셜 로그인 (Social Login - Google)
사용자의 가입 허들을 낮추고 안전한 인증 처리를 위해 OAuth 2.0 기반의 구글 소셜 로그인을 도입했습니다.

### ⚙️ 어떻게 구현했는지 (How)
* **Spring Security OAuth2 연동:** Spring Security의 `OAuth2UserService`를 활용하여 OAuth2 인증 흐름(인가 코드 발급, 토큰 교환 등)을 프레임워크 레벨에서 자동으로 처리하도록 구성했습니다.
* **JWT 기반 Stateless 인증:** 소셜 인증이 완료된 후, FACET 서버만의 자체적인 JWT(Json Web Token)를 발급하여 클라이언트에게 전달합니다. 이후의 모든 API 요청은 `@AuthenticationPrincipal`을 통해 인가된 사용자만 접근할 수 있도록 제어합니다.

### 💡 왜 이렇게 했는지 (Why)
* **인증 책임의 위임 (Delegation):** 인증 프로세스와 비밀번호 관리 등 민감한 보안 책임을 Google과 같은 신뢰할 수 있는 OIDC 공급자에게 위임했습니다.
* **비즈니스 로직 집중:** OAuth2의 복잡한 통신 규격을 Spring Security에 맡김으로써, 백엔드 개발자는 인증 완료 후 넘어온 유저 정보를 우리 DB 규격에 맞게 정규화하고 비즈니스 로직을 처리하는 데에만 집중할 수 있었습니다.

## 3. 🔑 안전한 비밀번호 변경 (Secure Password Change)
로그인된 사용자가 본인 확인을 거쳐 새로운 비밀번호로 안전하게 변경하는 프로세스입니다.

### ⚙️ 어떻게 구현했는지 (How)
* **요청 및 식별:** 클라이언트가 '현재 비밀번호'와 '새 비밀번호'를 POST 요청으로 보내면, 서버는 프론트가 넘겨주는 유저 ID를 믿지 않고 **JWT 토큰 내부의 정보(`@AuthenticationPrincipal`)를 추출**하여 요청의 주체를 직접 식별합니다.
* **본인 확인 (단방향 암호화 대조):** `passwordEncoder.matches(입력한 현재PW, DB의 해시값)`를 통해 현재 비밀번호가 맞는지 검증합니다. 불일치 시 `INVALID_PASSWORD` 예외를 발생시키고 즉시 로직을 중단합니다.
* **새 비밀번호 암호화 저장:** 본인 확인이 완료되면, 새 비밀번호를 `passwordEncoder.encode()`로 단방향 암호화하여 DB에 업데이트합니다.

### 💡 왜 이렇게 했는지 (Why)
* **주체 위변조 방지:** "누구의 비밀번호를 바꿀 것인가"에 대한 판단을 클라이언트의 파라미터가 아닌, 서버가 발급한 불변의 토큰(JWT)에 의존함으로써 타인 계정의 탈취 및 수정 시도를 차단했습니다.
* **평문 저장 방지 원칙:** 사용자의 비밀번호는 그 어떤 순간에도 서버나 로그에 평문으로 남지 않도록 스프링 시큐리티의 `PasswordEncoder`를 통한 강력한 해싱 처리를 강제했습니다.

## 4. 👤 유저 정보 조회 및 수정 (Profile Management)
사용자의 개인정보(주소, 닉네임, 연락처 등)를 안전하게 관리합니다.

### ⚙️ 어떻게 구현했는지 (How)
* 토큰 검증 필터를 통과한 정상적인 유저만 마이페이지 정보에 접근할 수 있도록 API 엔드포인트 접근 권한을 제한했습니다.
* 유저 정보 업데이트 시, JPA의 더티 체킹(Dirty Checking) 혹은 명시적인 Update 메서드를 활용하여 변경된 필드만 안전하게 데이터베이스에 반영합니다.

### 💡 왜 이렇게 했는지 (Why)
* **데이터 접근 통제:** 유저 식별을 전적으로 JWT 기반 Context에 맡겨, 인가되지 않은 사용자가 URL 파라미터 조작 등으로 다른 사람의 개인정보를 열람하거나 수정하는 수직적 권한 상승(IDOR) 취약점을 방지했습니다.
