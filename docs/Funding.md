# 🎁 프리오더 펀딩 상세 시스템 기능 명세

> FACET의 핵심 비즈니스 로직인 '조건부 예약 펀딩'을 구현하며 마주한 기술적 챌린지와 해결 과정을 정리한 문서입니다. 대량의 데이터 처리부터 결제 무결성, 그리고 동시성 문제까지 백엔드 관점에서의 깊이 있는 고민을 담았습니다.

---

## 1. 🛍️ 상품 조회 및 동적 페이징 처리 (Product Display & Paging)
대량의 펀딩 프로젝트 데이터를 빠르고 직관적으로 제공하기 위해, JPA 기반의 최적화된 페이징 및 정렬 기능과 실시간 통계 계산을 구현했습니다.

### ⚙️ 어떻게 구현했는지 (How)
* **JPA 페이징 기법:** `PageRequest`를 활용해 메인 리스트는 9개, 상세 리스트는 4개씩 분할 조회하여 응답 속도를 높였습니다.
* **동적 정렬 (Dynamic Sorting):** 클라이언트의 요청 파라미터(`currentFilter`)에 따라 최신순(`idx`), 인기순(`percent`), 참여순(`supporters`)으로 `Sort` 객체를 동적으로 생성하여 정렬 기준을 적용합니다.
* **실시간 펀딩 달성률 계산:** 커스텀 쿼리를 작성하여 전체 데이터가 아닌 **'결제 완료(PAID)' 상태의 주문 데이터만 조회**해 모인 금액과 서포터 수를 즉각적으로 계산합니다.

### 💡 왜 이렇게 했는지 (Why)
* **네트워크 및 메모리 최적화:** 모든 상품 데이터를 프론트엔드로 한 번에 전송하면 응답 지연과 서버 메모리 부하가 발생합니다. 페이징 처리를 통해 꼭 필요한 데이터만 빠르게 전달하도록 설계했습니다.
* **데이터 정합성 보장:** 서포터 수나 누적 금액을 상품 테이블에 따로 저장(역정규화)하는 대신, 조회 시점에 결제 완료된 데이터베이스 레코드만 취합하는 방식을 택했습니다. 이를 통해 결제 취소나 상태 변경 시 발생할 수 있는 데이터 불일치 문제를 원천적으로 차단했습니다.

## 2. 💳 펀딩 결제 및 3단계 무결성 검증
클라이언트 측의 데이터 위·변조를 방지하기 위해 PortOne SDK를 활용하며, 백엔드 중심의 꼼꼼한 결제 데이터 교차 검증을 수행합니다.



### ⚙️ 어떻게 구현했는지 (How)
결제 전 `create()` 메서드로 DB에 가주문을 먼저 생성하고, 결제 완료 후 `verify()`에서 아래 **3단계 교차 검증**을 거칩니다.
1. **유저 검증:** 요청을 보낸 유저 ID와 주문서에 기록된 유저 ID의 일치 여부를 확인합니다.
2. **리워드 검증:** 사용자가 요청한 리워드 IDX가 실제 DB에 존재하는지 필터링하여 검증합니다.
3. **금액 무결성 검증:** 백엔드에서 직접 계산한 총액 `(리워드 단가 * 수량) + 배송비`와, PortOne PG사에서 응답받은 실제 결제 금액이 정확히 일치하는지 대조합니다.

### 💡 왜 이렇게 했는지 (Why)
* **프론트엔드 조작 방지 (Zero-Trust 기반):** 브라우저 상에서는 개발자 도구를 통해 결제 금액이나 리워드 ID를 임의로 조작할 수 있습니다. 악의적인 '0원 결제'나 타인 계정 도용을 막기 위해, 모든 핵심 데이터는 **백엔드 DB 기준으로 재계산하여 PG사 데이터와 대조**하는 안전한 아키텍처를 채택했습니다.

## 3. 🔒 동시성 제어 및 비동기 통신
사용자가 몰리는 결제 오픈 상황에서 발생할 수 있는 '초과 결제(오버부킹)'를 원천 방지하고, 외부 API 통신 지연을 최소화하여 서버 안정성을 확보합니다.



### ⚙️ 어떻게 구현했는지 (How)
* **비관적 락(Pessimistic Lock) 적용:** 리워드 재고 조회 시 `@Lock(LockModeType.PESSIMISTIC_WRITE)`를 사용하여 DB 레벨의 배타적 락을 걸었습니다. 데드락(Deadlock) 방지를 위해 타임아웃은 3,000ms로 설정했습니다.
* **CompletableFuture 비동기 처리:** PortOne PG사에 결제 내역을 조회하거나, 금액 불일치로 결제 취소를 요청할 때 `CompletableFuture`를 활용하여 비동기로 전환했습니다.
* **안전한 롤백(Rollback) 처리:** 검증 과정에서 데이터가 일치하지 않으면 주문 상태를 즉시 `CANCELLED`로 변경하고, PG사 서버에 비동기로 취소 사유(`amount-mismatch-error`)를 포함한 결제 취소 요청을 전송합니다.

### 💡 왜 이렇게 했는지 (Why)
* **재고 갱신 손실(Lost Update) 방지:** 인기 리워드의 경우 다수의 사용자가 동시에 결제를 시도하면, 동일한 재고 데이터를 읽고 수정하여 마이너스(-) 재고가 발생할 위험이 있습니다. DB 단에서 비관적 락을 걸어 순차적으로 재고 차감(`reward.reduceQuantity()`)이 이루어지도록 강제하여 무결성을 지켰습니다.
* **서버 스레드 블로킹 방지:** 외부 PG사 서버와의 통신은 응답 지연(Latency)이 발생할 수 있습니다. 이를 동기적으로 처리하면 WAS의 스레드가 묶여 전체 서비스 장애로 이어질 수 있으므로, 비동기 처리를 도입하여 서버 자원 활용률과 응답성을 크게 높였습니다.
